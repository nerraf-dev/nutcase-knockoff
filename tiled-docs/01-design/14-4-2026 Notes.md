For v1, keep it as a single executable and stop widening scope.
An exe plus a loose `controller` folder makes the game feel more like a toolchain than a finished party game. For something small, social, and casual, single-file distribution is the right product choice even if the implementation is a bit uglier for now. 

Treat the inline-font fix as an **acceptable** temporary compromise in service of that constraint. It is not elegant, but it preserves the simpler user experience, which matters more at this stage. Make sure documentation is clear!

Focus priority: ship one family-friendly reveal game that works, feels good, and is easy to share. 
- Stable controller flow
- enough content to avoid feeling thin
- clean onboarding path
- confidence from testing. 
Put it on itch.io -  give it away. Aim to use real players to direct the next move. 

Recommended roadmap:
1. `Tiled v1`
Make one polished family-friendly game type, add content, test it properly, ship it.

2. `Tiled v1.x`
Add authoring discipline before new mechanics: question-set structure, content validation, maybe basic user-created packs if they fit the existing flow cleanly.

3. `Tiled v2`
Add 1 or 2 new reveal-based game types, but only if they reuse enough of the host/controller/session architecture to justify staying in the same product.


For the alternative version: model it as a content-layer fork or mode toggle first unless the mechanics truly diverge. If it needs different moderation, tone, scoring, hosting assumptions, or player expectations, then split it later.

For the educational angle: "*sounds like a separate product framing, not just a feature set.*" 
Prove the host/controller/content architecture on the party game, then spin a constrained classroom variant from the same core once you know which parts are genuinely reusable.
